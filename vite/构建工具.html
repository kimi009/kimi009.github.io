构建工具 #

不用关心代码如何编译，如何打包，只需要关心自己的业务代码，剩下的都由它来做的事

构建工具分类 #

webpack、vite、esbuild、rollup、parcel、grant、gulp、

浏览器只认识html,css js

企业级项目可能具备的功能

1、typescript 2、react/vue 3、sacss/less/postcss/component-style ：安装一系列loader等编译工具 4、语法降级：babel-->将es语法转化为旧版本浏览器支持的语法 5、体积优化压缩： uglifyjs--->压缩代码

构建工具支持的能力 #

构建工具是不需要用户关心代码如何在浏览器执行，只需要关注业务。构建工具会通过配置文件自动编译成浏览器可以运行的代码

1、模块化开发支持 #

多模块支持，commonjs esmodule

支持从node_modules里面引入代码，浏览器默认是不认识node_modules

2、处理代码兼容性处理 #

babel降级, less、ts语法转化（构建工具集成了这些语法转化工具）

3、提高项目性能 #

压缩代码，代码分割

4、优化开发体验 #

HMR（热更新 Hot module replacement）

5、开发服务器 #

解决跨域问题（vue-cli、react-cli)

vite相比webpack的优势 #

在浏览器支持 ES 模块之前，JavaScript 并没有提供原生机制让开发者以模块化的方式进行开发。这也正是我们对 “打包” 这个概念熟悉的原因：使用工具抓取、处理并将我们的源码模块串联成可以在浏览器中运行的文件。

时过境迁，我们见证了诸如 webpack、Rollup 和 Parcel 等工具的变迁，它们极大地改善了前端开发者的开发体验。

然而，当我们开始构建越来越大型的应用时，需要处理的 JavaScript 代码量也呈指数级增长。包含数千个模块的大型项目相当普遍。基于 JavaScript 开发的工具就会开始遇到性能瓶颈：通常需要很长时间（甚至是几分钟！）才能启动开发服务器，即使使用模块热替换（HMR），文件修改后的效果也需要几秒钟才能在浏览器中反映出来。如此循环往复，迟钝的反馈会极大地影响开发者的开发效率和幸福感。

Vite 旨在利用生态系统中的新进展解决上述问题：浏览器开始原生支持 ES 模块，且越来越多 JavaScript 工具使用编译型语言编写。

  • vite一开始就启动开发服务器，不需要解析所有的模块，而是按需加载
  • webpack一开始就要把所有的模块加载，转换，解析所有的依赖，所以慢
